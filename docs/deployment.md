# Deployment — Render

Alpha E-com runs on [Render](https://render.com) as a single Node.js web service.
The Express server hosts **both the API and the static frontend** from the same URL,
so no separate CDN or frontend host is required.

---

## Architecture on Render

```
Browser
  └─→  https://alpha-ecom-api.onrender.com
          │
          ├── GET /              → Express static → frontend/index.html
          ├── GET /css/*.css     → Express static → frontend/css/
          ├── GET /js/*.js       → Express static → frontend/js/
          │
          ├── GET  /api/products → Express routes → MongoDB Atlas
          ├── POST /api/cart     → Express routes → MongoDB Atlas
          ├── POST /api/orders   → Express routes → MongoDB Atlas
          └── GET  /health       → { status: 'ok', env: 'production' }
```

MongoDB is **not** hosted on Render. Use the free [MongoDB Atlas](https://www.mongodb.com/atlas) M0 cluster.

## One-time setup

### 1. Create a MongoDB Atlas cluster

1. Sign up at <https://www.mongodb.com/atlas> → create a free **M0** cluster.
2. **Database Access** → Add user → username + strong password (record both).
3. **Network Access** → Add IP → `0.0.0.0/0` (allow from anywhere — Render uses dynamic IPs).
4. **Connect** → "Connect your application" → copy the connection string:
   ```
   mongodb+srv://<user>:<password>@<cluster>.mongodb.net/alpha-ecom?retryWrites=true&w=majority
   ```
   Replace `<user>`, `<password>`, and `<cluster>` with your values.

### 2. Connect the GitHub repo to Render

1. Go to <https://dashboard.render.com> → **New** → **Blueprint**.
2. Connect your GitHub account and select the `alpha-ecom` repository.
3. When prompted for the **Render YAML path**, enter:
   ```
   backend/render.yaml
   ```
   (or move `backend/render.yaml` to the repo root to use the default path)
4. Render will detect the `alpha-ecom-api` web service from the YAML and show a preview.
5. Click **Apply**.

### 3. Set required environment variables

After Render creates the service, go to:
**Dashboard → alpha-ecom-api → Environment**

Set the following variables (those marked `sync: false` in `render.yaml` are not set automatically):

| Variable | Value | Notes |
|---|---|---|
| `MONGO_URI` | `mongodb+srv://...` | From Atlas step above |
| `CORS_ORIGIN` | `https://alpha-ecom-api.onrender.com` | Your Render service URL |

`JWT_SECRET` is auto-generated by Render (via `generateValue: true`) — no action needed.
`NODE_ENV` is set to `production` automatically from `render.yaml`.
`PORT` is injected by Render — the server reads it automatically.

**How to set a variable:**
1. Dashboard → select service → **Environment** tab.
2. Click **Add Environment Variable**.
3. Enter the key and value → **Save Changes**.
4. Render triggers an automatic redeploy.

### 4. Verify the deployment

Once the deploy finishes (green **Live** badge):

```bash
# Health check
curl https://alpha-ecom-api.onrender.com/health
# → {"status":"ok","env":"production"}

# API check
curl https://alpha-ecom-api.onrender.com/api/products
# → {"success":true,"data":[...]}

# Frontend
open https://alpha-ecom-api.onrender.com
```

## Triggering a manual redeploy

**From the Render Dashboard:**
1. Dashboard → **alpha-ecom-api** → **Deploys** tab.
2. Click **Manual Deploy** → **Deploy latest commit**.

**From the Render CLI** (install with `npm i -g @render/cli`):
```bash
render deploys create <service-id>
```

**Automatic deploys:** every push to `main` on GitHub triggers a deploy automatically
(configured via the Render → GitHub integration in the service settings).

---

## Rolling back to a previous deploy

1. Dashboard → **alpha-ecom-api** → **Deploys** tab.
2. Find the deploy you want to roll back to (each row shows commit SHA + timestamp).
3. Click the **⋯** menu on that row → **Rollback to this deploy**.
4. Confirm — Render immediately re-deploys that exact build without re-running the build step.

> **Note:** Rolling back the app does not roll back the database. If a new deploy ran
> a schema migration or destructive data operation, you must restore from a MongoDB Atlas
> backup separately (Atlas → Clusters → … → Restore).

## Environment variables reference

| Variable | Required | Description |
|---|---|---|
| `PORT` | Auto | Injected by Render — do not set manually |
| `NODE_ENV` | Yes | `production` (set in `render.yaml`) |
| `MONGO_URI` | Yes | MongoDB Atlas connection string |
| `JWT_SECRET` | Yes | Auto-generated by Render; override if needed |
| `CORS_ORIGIN` | Yes | Comma-separated allowed origins — set to Render URL |

## Free tier caveats

Render's free plan spins down web services after **15 minutes of inactivity**.
The next request after a cold start takes ~30 seconds while the service wakes up.

To avoid spin-down:
- Upgrade to the **Starter** plan (~$7/month).
- Or use an external uptime monitor (e.g. [UptimeRobot](https://uptimerobot.com)) pinging
  `/health` every 10 minutes to keep the service warm.
